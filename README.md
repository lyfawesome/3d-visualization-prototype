# 3D Visualization Prototype

This repository contains a Windows desktop prototype for structural test-bench model visualization and first-stage CalculiX preprocessing/postprocessing.

## Goal

The prototype demonstrates these functions:

- Load specimen model files generated by professional 3D/CAD software.
- Create/open a project folder that owns imported inputs, mesh files, solver files, results, and exports.
- Automatically create an interactive 3D visualization scene.
- Read a configured load-point file.
- Pick STEP/OCCT constraint faces before volume meshing.
- Generate a Netgen tetrahedral volume mesh with boundary markers for selected constraint faces.
- Guide the user to pick load application nodes on the generated boundary mesh.
- Export and run a CalculiX solid tetrahedral input deck.
- Read CalculiX `.frd` displacement results and display displacement magnitude on the generated boundary mesh.

## Technical Stack

- C++17
- Qt 6 Widgets for the Windows desktop interface
- VTK for 3D rendering, interaction, labels, color mapping, and screenshot export
- OpenCASCADE / OCCT for direct STEP/STP CAD import
- Netgen for STEP-derived tetrahedral volume meshing, using either `netgen.exe` or the official prebuilt `netgen-mesher` Python wheel
- CalculiX `ccx` for finite element solving
- CMake and MSYS2 UCRT64 for Windows dependency deployment with prebuilt binary packages

Supported model formats:

- STEP/STP, read through OCCT `STEPControl_Reader`
- STL, read through VTK
- OBJ, read through VTK

## Build on Windows

Prerequisites:

- Windows 10/11
- MSYS2 installed at `C:\msys64`
- Git
- PowerShell

Run:

```powershell
Set-ExecutionPolicy -Scope Process Bypass
.\scripts\build_windows.ps1
```

The script installs the required MSYS2 UCRT64 binary packages if they are missing, then configures CMake and builds the Release executable with Ninja.

Core MSYS2 build packages:

- `mingw-w64-ucrt-x86_64-toolchain`
- `mingw-w64-ucrt-x86_64-cmake`
- `mingw-w64-ucrt-x86_64-ninja`
- `mingw-w64-ucrt-x86_64-eigen3`
- `mingw-w64-ucrt-x86_64-exprtk`
- `mingw-w64-ucrt-x86_64-nlohmann-json`
- `mingw-w64-ucrt-x86_64-qt6-base`
- `mingw-w64-ucrt-x86_64-vtk`
- `mingw-w64-ucrt-x86_64-opencascade`

CalculiX is required for solving, not for compiling the C++ targets:

- `mingw-w64-ucrt-x86_64-calculix-ccx`

If a restricted environment prevents GCC from writing temporary files under `C:\msys64\tmp`, use a writable workspace temp directory before invoking CMake:

```powershell
New-Item -ItemType Directory -Force .tmp\msys2 | Out-Null
$env:TMP = "$PWD\.tmp\msys2"
$env:TEMP = "$PWD\.tmp\msys2"
```

Netgen is discovered in this order:

- `NETGEN_EXE`
- application directory `netgen.exe`
- `extern\netgen\bin\netgen.exe`
- `PATH`
- `C:\msys64\ucrt64\bin\netgen.exe`
- `NETGEN_PYTHON`
- `extern\netgen\python-env\Scripts\python.exe`

The MSYS2 UCRT64 package repository used during development did not provide a `mingw-w64-ucrt-x86_64-netgen` package, so Netgen is supplied separately.

To download the official prebuilt Netgen Python wheel and install it into the local ignored `extern\netgen` environment:

```powershell
.\scripts\download_netgen_windows.ps1
```

The script downloads binary wheels for `netgen-mesher` and its dependencies, then creates `extern\netgen\python-env`. It does not clone or compile Netgen source code. The C++ application calls `scripts\netgen_mesh_step.py` in a separate Python process to write Netgen `.vol` output, then parses `.vol` and generates VTK and CalculiX data in C++.

Check Netgen discovery with:

```powershell
.\scripts\check_netgen.ps1
```

Run the prototype:

```powershell
.\scripts\run_windows.ps1
```

- `CMakePresets.json`
- `README.md`

Dependency policy:

- MSYS2 dependencies are installed on each machine and are not copied into Git. Run `.\scripts\build_windows.ps1` on a new Windows machine.
- Netgen is not committed. Install or copy it into the ignored `extern\netgen\` location, then verify with `.\scripts\check_netgen.ps1`.
- CalculiX is expected from MSYS2 at `C:\msys64\ucrt64\bin\ccx.exe` or from `PATH`; do not commit `ccx.exe`.
- vcpkg source trees and installed packages are not committed. Keep `vcpkg.json`, `vcpkg-overlays/`, and scripts in Git; recreate dependencies locally.

For the verified `windows-msys2-ucrt64` build route, the only local dependency folder that normally needs to be copied between machines is:

```text
extern/netgen/
```

Do not copy these folders for the MSYS2 build route:

```text
build/
vcpkg_installed/
extern/vcpkg/
extern/prebuilt/
p/
s/
```

Before pushing, stage from an explicit allow-list instead of using `git add .`:

```powershell
git add src scripts docs samples CMakeLists.txt CMakePresets.json README.md agent.md vcpkg.json vcpkg-overlays .gitignore
git status --short
```

If files were staged accidentally with `git add .`, unstage them without losing local edits:

```powershell
git reset
```

## Current Workflow

1. Start the application.
2. Use `New Project` to select/create a project folder, or `Open Project` to open an existing `project.vpproj`.
3. Use `Open Model` to import a STEP/STP model into the project `models/` folder for the full finite element workflow. STL/OBJ remain visualization inputs and do not carry CAD face semantics.
4. Enable `Pick Constraint Faces` and click one or more model faces. Each picked triangle selects its full source OCCT face. The first version writes selected faces as fully fixed `UX/UY/UZ=0` constraints.
5. Use `Generate Volume Mesh` to create the Netgen tetrahedral volume mesh and boundary-surface VTK output.
6. Use `Load Points` to import `loading_points.json` into the project `inputs/` folder.
7. Enable `Pick Load Points` and click the generated boundary surface. The selected loading device is mapped to the nearest mesh boundary node.
8. Use `Export INP` to write `solver/model.inp`, or `Run CalculiX` to write `solver/model.inp`, run `ccx` in `solver/`, copy `.frd/.dat` to `results/`, and load the resulting displacement contour on the generated boundary mesh.
9. Use `Open Result` to load `solver/model.frd` or `results/model.frd` after a volume mesh is available.

Project folder layout:

- `project.vpproj`: project metadata.
- `models/`: imported model files.
- `inputs/`: imported loading-device JSON files.
- `meshes/netgen/`: Netgen `.vol` input/output and generated `.vtk` volume/boundary output.
- `solver/`: CalculiX working directory, including `model.inp`.
- `results/`: copied CalculiX `.frd/.dat` result files.
- `exports/`: screenshots and future user exports.

Current CalculiX export uses the generated volume mesh as the solver source. It writes `C3D4` tetrahedral elements, a default solid material/section, fully fixed boundary nodes from selected constraint face markers, and concentrated nodal loads from mapped mesh load nodes. Export and solve are blocked until a project is open, a tetrahedral volume mesh exists, at least one constraint face exists, and at least one mapped mesh load node is available.

The internal mesh data structure is `VolumeMesh`; it stores Netgen nodes, `C3D4` tetrahedra, boundary faces, and constraint-marker node sets.

CalculiX `.frd` displacement postprocessing maps result node ids back to generated mesh node ids and colors the generated boundary surface with displacement magnitude. It no longer maps solver results onto the original STEP render mesh.

Run the Netgen backend smoke test:

```powershell
build\windows-msys2-ucrt64\NetgenStepMeshSmoke.exe samples\solid.step solver_runs\netgen_smoke 1
```

If neither `netgen.exe` nor the prebuilt wheel backend is configured, this smoke test prints a skipped message. Once Netgen is available, it validates the direct Netgen volume-mesh path and checks that the requested constraint marker produces boundary nodes.

For Netgen-generated meshes, constraint markers are not assumed to match Netgen's own surface numbering. The backend remaps Netgen boundary triangles back to the selected STEP/OCCT faces by geometric distance before writing direct Netgen boundary markers.
